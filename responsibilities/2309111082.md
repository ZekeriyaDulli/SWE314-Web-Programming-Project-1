## Responsibilities

### Database Schema Design
- Designed the complete MySQL relational schema for the `movie_archive` database — 16 tables in total
- Core entities: `users`, `shows`, `genres`, `actors`, `directors`, `seasons`, `episodes`, `watchlists`, `watchlist_items`, `ratings`, `watch_history`
- Extended entities (Phase 2): `tags`, `show_tags`, `collections`, `collection_shows`
- Defined 4 explicit many-to-many junction tables: `show_genres`, `show_actors`, `show_directors`, `show_tags`
- Applied appropriate constraints throughout: `UNIQUE` on `imdb_id`, `email`, and actor/director `full_name`; `CHECK (rating BETWEEN 1 AND 10)`; composite unique constraints on `(show_id, season_number)` and `(season_id, episode_number)` to prevent duplicate season/episode records
- Cascade deletes on all foreign keys to ensure referential integrity when users or shows are removed

### Stored Procedures — Show & Genre Operations
- Wrote `sp_get_all_shows(p_genre_id, p_min_year, p_max_year, p_min_rating, p_search)` — the primary catalogue query; uses LEFT JOINs to `show_genres`, `ratings` and `watch_history`; computes `platform_avg` and `rating_count` inline; all filter params nullable with `IF(p_param IS NULL, TRUE, condition)` guards
- Wrote `sp_get_show_detail(p_show_id, p_user_id)` — returns a single show row plus `is_watched` derived from `watch_history`
- Wrote `sp_add_show(p_imdb_id, OUT p_show_id)` — uses `INSERT IGNORE` on `imdb_id` so re-uploading the same IMDb ID is safe; returns the new or existing `show_id` via OUT param
- Wrote `sp_get_all_genres()`, `sp_get_show_genres(p_show_id)`, `sp_get_show_actors(p_show_id)`, `sp_get_show_directors(p_show_id)`, `sp_get_show_tags(p_show_id)`, `sp_get_show_seasons(p_show_id)`, `sp_get_season_episodes(p_season_id)`

## Key Technical Decisions

- **`INSERT IGNORE` on junction tables**: makes every sync and every watchlist add idempotent — calling the same procedure twice produces the same state, with no duplicate-key errors surfacing to the application layer
- **Nullable filter params in `sp_get_all_shows`**: a single stored procedure handles all filter combinations; the backend passes `NULL` for unset filters rather than building dynamic SQL
- **Composite UNIQUE constraints on seasons and episodes**: enforces data integrity at the database level so the OMDb sync's `ON DUPLICATE KEY UPDATE` pattern works correctly without application-level duplicate checks
- **Cascade deletes**: removing a user automatically cleans up their watchlists, ratings, and watch history; removing a show cleans up all relationship tables — no orphan rows
