### Models & Schemas
- Wrote `models.py` ŌĆö 16 `SQLModel` table classes mapping exactly to the MySQL schema; no relationships or back-populates defined (the service layer uses stored procedures, not ORM joins)
- Wrote `schemas.py` ŌĆö strict separation between DB models and API contracts:

### Service Layer (`services.py`)
- Implemented all business logic as flat functions ŌĆö no classes; routes call one function and return the result
- All database access via `sqlalchemy.text("CALL sp_name(:p)")` ŌĆö the 7 stored procedures are treated as the data API; no raw SQL built in Python
- OUT param pattern: `session.execute(text("CALL sp_name(:p, @out)"), {...})` followed by `SELECT @out` to retrieve the generated ID
- `register_user`: hashes password with `passlib CryptContext(schemes=["bcrypt"])`; detects duplicate email by checking `SELECT @out IS NULL`; raises `HTTPException(409, "An account with this email already exists.")`
- `login_user`: calls `sp_get_user_by_email`; raises `HTTPException(401)` on missing user or failed `passlib.verify()`; creates JWT with `python-jose` encoding `{"sub": user_id, "exp": ...}`
- `process_csv_upload`: decodes bytes, skips blank lines and header, validates each line against `^tt\d+$`; calls `sp_add_show` per valid ID; returns `{"imported": N, "skipped": M}` summary

### Background Sync & YouTube Trailer Scraping
- Implemented `run_full_sync(session, client, missing_only=False)`:
  - Protected by `asyncio.Lock()` ŌĆö concurrent sync requests are rejected with 400
  - Progress tracked in module-level `_sync_state` dict; frontend polls `GET /admin/sync/status` every 2 seconds
  - `missing_only=True` filters the query to `WHERE title IS NULL OR imdb_rating IS NULL OR poster_url IS NULL OR trailer_url IS NULL` ŌĆö drastically reduces sync time on large catalogues
  - Per-show `try/except Exception: session.rollback()` ensures one failed title never aborts the entire sync
  - For TV series, calls `_sync_tv_seasons` which upserts seasons and episodes with `ON DUPLICATE KEY UPDATE` ŌĆö re-running the sync refreshes episode data without creating duplicates
- Implemented `_fetch_youtube_trailer(title, year, client)` ŌĆö no YouTube API key required:
  - GETs `youtube.com/results?search_query={title} ({year}) official trailer` with `Accept-Language: en-US` header
  - Targets `"videoRenderer":{"videoId":"..."` regex ŌĆö this renderer is exclusive to organic search results; ads use `promotedSparklesWebRenderer` and are never matched
  - Stores only the 11-character video ID in `trailer_url VARCHAR(20)`

### Routing Layer (8 Router Files)
- All routes follow the thin-route rule: inject session, call one service function, return result ŌĆö 3ŌĆō5 lines per handler
- `routers/external.py`: poster proxy (`GET /poster/{imdb_id}`) streams images via `StreamingResponse` with `Cache-Control: public, max-age=604800`; sync endpoints return `202 Accepted` with background task; `POST /admin/sync/start-missing` passes `missing_only=True`
- `routers/users.py`: register returns `201`; login returns `TokenResponse`
- `routers/ratings.py`: upsert returns `201`; delete returns `204 No Content`
- `routers/watchlists.py`: create returns `201`; get detail raises `404` if watchlist not found or not owned; delete item returns `204`
- `dependencies.py`: `get_current_user` decodes JWT and fetches user; raises `401` on any failure; `require_admin` raises `403` if `user_id != settings.admin_user_id`

## Key Technical Decisions

- **Fat services / thin routes**: all business logic, permission checks, and SQL in `services.py`; routes are ignorant of SQL ŌĆö this made unit testing and debugging dramatically easier
- **Stored procedures as the data API**: the 7 MySQL stored procedures handle all write-path business logic; read operations use raw `sqlalchemy.text()` queries directly
- **`missing_only` sync mode**: the YouTube Data API allows only 100 searches per day; scraping `youtube.com/results` directly has no quota limit; the missing-only filter reduces per-sync scraping to only titles that actually need it
- **Proxy poster endpoint**: the OMDb API key is never sent to the browser; all image requests go through `/poster/{imdb_id}` which fetches and streams the image server-side
